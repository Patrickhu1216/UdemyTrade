# Claude Code 精髓的關鍵地圖：  接下來的學習地圖與核心功能

### **提示詞與上下文工程（Prompt & Context Engineering）：**
學習如何為 Claude Code 撰寫精準的提示詞，以及如何餵給它「剛剛好」的上下文，讓 AI 不會因為資訊過載而失焦，或因為資訊不足而瞎猜。

### **深入計劃模式（Plan Mode）：**
這個模式在前面的命令列參數中也有提到（--permission-mode plan）。在 Plan Mode 下，AI 在動手改 code 之前會先跟你討論、列出完整的架構規劃，非常適合大範圍的重構或新功能開發。 

### **記憶管理（Memory Management）：**
理解 Claude Code 如何在長對話中管理、壓縮或保留記憶，避免隨著對話變長而「金魚腦」，這對於大型專案的連續開發至關重要。  

### **子代理人與特化技能（Sub-agents & Agent Skills）：**
探討 Claude Code 如何在背後調度其他「小幫手（Sub-agents）」來各司其職，以及它如何運用特定的技能（Skills）來解決複雜的工程問題。  

### **MCP 伺服器（Model Context Protocol）：**
這就是前面提到的 /mcp 指令延伸。MCP 是讓 AI 連結外部工具、資料庫或自訂 API 的強大橋樑，能讓 Claude Code 的超能力跨出本地端專案。

# 提示詞與上下文工程（Prompt & Context Engineering

大型語言模型（LLM）協作時的核心真理：「Garbage in, Garbage out」

## 核心黃金公式：優質輸入 ＝ 優質輸出

雖然現在的 AI 非常聰明，即使你給它模糊、隨性的爛提示詞，它偶爾也能通靈交出「還湊合」的答案。但如果你想要 AI 產出穩定、高品質、符合生產環境標準的程式碼，提供精準的輸入才是關鍵。兩個黃金元素組成：

## 1. 明確的指令（Specific Instructions）
* **做法：**用清晰、毫無歧義的文字，具體描述你希望 Claude Code 執行的任務或解決的問題。 
* **心法：**把 AI 當成一個剛進團隊、技術很好但完全沒有專案背景知識的實習生。告訴它「要做什麼（What）」以及「遵循什麼標準（How）」，而不是只丟一句「幫我修這個 Bug」。

## 2. 精準的上下文（Relevant Context）
* **做法：**提供與當前任務絕對相關的額外資訊、檔案內容或錯誤 log。 
* **核心警告（避免資訊過載）：**很多人以為把整個專案的程式碼或無關的設定檔通通餵給 AI 是最好的，但實際上，提供不必要的冗餘資訊（Unnecessary Information）反而會導致輸出品質下滑。AI 可能會因此抓錯重點、被不相關的程式碼干擾，甚至產生幻覺。 

### 精準控管上下文，就是幫 AI 減輕大腦負擔。

好的脈絡（Context）不是越多越好，而是要「精準、乾淨」。只給 Claude Code 真正需要關聯的檔案，它的邏輯推理能力才會發揮到極致。這也是為什麼在接下來的專案實戰中，學會如何「限縮 AI 視線」會是一個非常重要的核心技巧！

# 實戰：從規格書（Specification）啟動專案與上下文工程

當我們要從頭建立一個新應用程式（或接手既有專案）時，最完美的起手式就是提供一份規格書檔案（Specification Document），讓 AI 完整理解整個專案的靈魂與技術堆疊。

## 1. 實戰專案：富文本雲端筆記 App
* **核心功能：** 富文本編輯器（Text Formatting）、筆記瀏覽/編輯/刪除、公開分享連結（允許非擁有者檢視）。  
* **技術堆疊（Tech Stack）：** Next.js + Bun + TypeScript + Tailwind CSS。  
* **核心套件：** Better-Auth（第三方認證）、Tiptap（富文本編輯器）、SQLite（輕量生產/開發資料庫）。  

<br>

## 2. 核心核心心法：如何進行高品質的「上下文工程」？

1. 使用 @ 符號精準鎖定檔案在提示詞中輸入 `@` 符號（例如 `@spec.md`），是 Claude Code 官方指定的檔案關聯方式。 

* 金牌法則：Claude Code 具備極強的尋找檔案能力，你不需要把專案所有檔案通通 @ 進來。**「只餵入絕對相關的檔案」**才能防止上下文過載，確保輸出品質。

2. 使用 XML 標籤包裹外部文字當需要從外部複製官方文件（如 Better-Auth 的資料庫 Schema 規範）並貼入終端機作為額外脈絡時，講師強烈建議使用 XML 標籤（例如 `<auth_doc> ... </auth_doc>`）將其包覆起來。

* 好處：這能幫助大語言模型（LLM）極其精準地識別「指令」與「參考資料」的邊界，大幅提升生成準確度。  

<br>

## 3. 實戰操作流程拆解

### 第一步：建立原始規格書（SPEC.md）
1. 先在 ChatGPT 中以白話文描述專案需求，請其生成一份結構完整的「技術規格書」。  
2. 在專案根目錄建立 `SPEC.md` 檔案，並將內容貼上（此時格式可能較為混亂）。  

### 第二步：餵入指令與外部脈絡，啟動 Claude Code

CLI：

```Plaintext
We're building an app described in @spec.md . 
Please format this file as proper Markdown.

Also update the file and update the part about the users table and auth-related tables. We're using the betterauth library, which expects a certain database structure:

<betterauth_docs>
[此處貼上從 Better-Auth 官網複製的 Markdown 資料庫 Schema 文件]
</betterauth_docs>
```

### 第三步：切換加速模式並執行
在送出提示詞前，按下快捷鍵 SHIFT + TAB 切換到 「自動接受修改模式（Accept Edits Mode）」，然後按下 Enter。Claude Code 隨即自動讀取 Spec.md、消化 Better-Auth 的文件，並完美重構了整份規格書。

### 知識圖解：規格書重構前後對比
```
[原始 ChatGPT 規格書] ───> (偵測到 Users 複數資料表，與 Better-Auth 衝突)
                                      │
                                      ▼ <加上 Better-Auth 官方文件脈絡>
[Claude Code 自動重構] ───> 1. 修正為 "user" 單數資料表與對應欄位
                            2. 完成精美的 Markdown 格式化
```

# Note: Prompt & Content Engineering Recommendation

## 1. 簡潔且精準（Concise & Precise）
* **核心心法：** 保持提示詞聚焦、乾淨。 
* **實作要點：** 你必須準確且清晰地描述希望 AI 解決的任務或問題，但請絕對避免塞入無關緊要的細節、廢話或無意義的客套修飾（Unnecessary Fluff）。  

## 2. 拒絕不必要的上下文（No Unnecessary Context）
* **核心心法：** 提供有用（Useful）的脈絡至關重要，但塞入無關的資訊只會適得其反。 
* **實作要點：**
    * 不要瞎猜檔案：只用 @ 標記你確定（KNOW）相關的檔案；不要因為「覺得（THINK）某個檔案可能有關係」就盲目把它 @ 進去。 
    * 過濾外部資料：無論是官方文件、報錯 Log 還是其他參考資訊，只要與當前任務沒有直接關係，一律不要提供，以免干擾 AI 的注意力。  

## 3. 先思考、再規劃、後提問（Think, Plan, Prompt）
* **核心心法：** 不要一邊敲鍵盤一邊想，更不要指望「等 AI 改錯了再來慢慢修」。
* **實作要點：** 
    * 動手前，先在腦中思考、建立清晰的實作計畫，最後才撰寫提示詞。
    * 如果你發現自己經常需要連續發送好幾條提示詞來修正、澄清 AI 的行為，這代表你前置的規劃不夠好。 
    * 對於所有非微不足道（Non-trivial）的複雜任務，請強烈建議啟動 「計畫模式（Plan Mode）」。  

## 4. 絕對不要「測試」AI（Don't "Test" The AI）
* **核心心法：** 你是要叫 AI 來幫你把工作搞定的，不要跟它玩捉迷藏。 
* **實作要點：** 
    * 如果你事先就知道這項任務存在某個技術難關、特定地雷或常見的踩坑點（Pitfall），請直接向 AI 攤牌，絕對不要對它隱瞞資訊。  
    * 在初始提示詞中，應直接指出已知的潛在錯誤，並附上你推薦的解決方案（Recommended Solution），引導 AI 直接繞過陷阱。  

## 5. 明確指定 AI 應該使用的工具（Explicitly Guide Tool Usage）
* **核心心法：** 不要一廂情願地「希望」AI 會自動聯想到正確的工具。
* **實作要點：** 
    * Claude Code 擁有強大的工具庫（例如底層的 Bash 執行、網頁請求，或透過 MCP 伺服器擴充的工具、子代理人 Sub-agents、特化技能等）。 
    * 如果你清楚知道某個任務（或任務的某個步驟）使用特定的工具或功能來做最合適，請在提示詞中明確下達指令指示它使用。

# 自主套件管理與 Claude Code 專案初始化（/init）

### 一、 為什麼「套件安裝」要親自動手？

- **避免 AI 的「時空錯亂」：**
AI 在自動安裝套件時，有時會傾向直接去修改 package.json 檔案，並在裡面填入舊版本的套件號碼，這會為專案埋下技術債。

- **確保最新穩定版本：**
身為開發者，你清楚知道目前需要哪些套件，親自使用套件管理工具（如 bun add）能 100% 保證安裝到最新、最正確的版本。

範例安裝的套件清單：
1. **better-auth：** 第三方身份驗證核心。  
2. **zod：** 用於嚴格驗證使用者輸入（User InputValidation）。  
3. **Tiptap 相關套件：** 遵循官方指南安裝富文本編輯器所需的依賴。  
4. **bun-types（開發依賴 -D）：** 避免 TypeScript 在處理 Bun 原生 API 時噴出型別錯誤。

<br>

### 二、 核心指令： `/init`

**CLI**

```Plaintext
/init
```

#### 1. `. /init` 到底做了什麼？

不論是全新的專案還是第一次引入 Claude Code 的既有老舊專案，`/init` 都會觸發 AI 對整棟代碼建築進行「深度安全檢查」與「架構通靈」：  

- 依賴分析：徹底掃描剛剛手動安裝的所有套件與版本。  
- 結構理解：翻遍專案內的所有資料夾與檔案，理解目前的架構設計。  
- 規格對齊：自動偵測並讀取專案中的 Spec.md，將技術規格與實際代碼進行交叉比對。  

#### 2. 最終產物：CLAUDE.MD 誕生

- 經過深度分析（期間會向你請求資料夾的探索權限），Claude Code 會在專案根目錄自動生成一個名為 CLAUDE.MD 的檔案。
