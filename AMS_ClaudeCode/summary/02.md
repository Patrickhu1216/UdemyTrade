# Claude Code 精髓的關鍵地圖：  接下來的學習地圖與核心功能

### **提示詞與上下文工程（Prompt & Context Engineering）：**
學習如何為 Claude Code 撰寫精準的提示詞，以及如何餵給它「剛剛好」的上下文，讓 AI 不會因為資訊過載而失焦，或因為資訊不足而瞎猜。

### **深入計劃模式（Plan Mode）：**
這個模式在前面的命令列參數中也有提到（--permission-mode plan）。在 Plan Mode 下，AI 在動手改 code 之前會先跟你討論、列出完整的架構規劃，非常適合大範圍的重構或新功能開發。 

### **記憶管理（Memory Management）：**
理解 Claude Code 如何在長對話中管理、壓縮或保留記憶，避免隨著對話變長而「金魚腦」，這對於大型專案的連續開發至關重要。  

### **子代理人與特化技能（Sub-agents & Agent Skills）：**
探討 Claude Code 如何在背後調度其他「小幫手（Sub-agents）」來各司其職，以及它如何運用特定的技能（Skills）來解決複雜的工程問題。  

### **MCP 伺服器（Model Context Protocol）：**
這就是前面提到的 /mcp 指令延伸。MCP 是讓 AI 連結外部工具、資料庫或自訂 API 的強大橋樑，能讓 Claude Code 的超能力跨出本地端專案。

# 提示詞與上下文工程（Prompt & Context Engineering

大型語言模型（LLM）協作時的核心真理：「Garbage in, Garbage out」

## 核心黃金公式：優質輸入 ＝ 優質輸出

雖然現在的 AI 非常聰明，即使你給它模糊、隨性的爛提示詞，它偶爾也能通靈交出「還湊合」的答案。但如果你想要 AI 產出穩定、高品質、符合生產環境標準的程式碼，提供精準的輸入才是關鍵。兩個黃金元素組成：

## 1. 明確的指令（Specific Instructions）
* **做法：**用清晰、毫無歧義的文字，具體描述你希望 Claude Code 執行的任務或解決的問題。 
* **心法：**把 AI 當成一個剛進團隊、技術很好但完全沒有專案背景知識的實習生。告訴它「要做什麼（What）」以及「遵循什麼標準（How）」，而不是只丟一句「幫我修這個 Bug」。

## 2. 精準的上下文（Relevant Context）
* **做法：**提供與當前任務絕對相關的額外資訊、檔案內容或錯誤 log。 
* **核心警告（避免資訊過載）：**很多人以為把整個專案的程式碼或無關的設定檔通通餵給 AI 是最好的，但實際上，提供不必要的冗餘資訊（Unnecessary Information）反而會導致輸出品質下滑。AI 可能會因此抓錯重點、被不相關的程式碼干擾，甚至產生幻覺。 

### 精準控管上下文，就是幫 AI 減輕大腦負擔。

好的脈絡（Context）不是越多越好，而是要「精準、乾淨」。只給 Claude Code 真正需要關聯的檔案，它的邏輯推理能力才會發揮到極致。這也是為什麼在接下來的專案實戰中，學會如何「限縮 AI 視線」會是一個非常重要的核心技巧！

# 實戰：從規格書（Specification）啟動專案與上下文工程

當我們要從頭建立一個新應用程式（或接手既有專案）時，最完美的起手式就是提供一份規格書檔案（Specification Document），讓 AI 完整理解整個專案的靈魂與技術堆疊。

## 1. 實戰專案：富文本雲端筆記 App
* **核心功能：** 富文本編輯器（Text Formatting）、筆記瀏覽/編輯/刪除、公開分享連結（允許非擁有者檢視）。  
* **技術堆疊（Tech Stack）：** Next.js + Bun + TypeScript + Tailwind CSS。  
* **核心套件：** Better-Auth（第三方認證）、Tiptap（富文本編輯器）、SQLite（輕量生產/開發資料庫）。  

<br>

## 2. 核心核心心法：如何進行高品質的「上下文工程」？

1. 使用 @ 符號精準鎖定檔案在提示詞中輸入 `@` 符號（例如 `@spec.md`），是 Claude Code 官方指定的檔案關聯方式。 

* 金牌法則：Claude Code 具備極強的尋找檔案能力，你不需要把專案所有檔案通通 @ 進來。**「只餵入絕對相關的檔案」**才能防止上下文過載，確保輸出品質。

2. 使用 XML 標籤包裹外部文字當需要從外部複製官方文件（如 Better-Auth 的資料庫 Schema 規範）並貼入終端機作為額外脈絡時，講師強烈建議使用 XML 標籤（例如 `<auth_doc> ... </auth_doc>`）將其包覆起來。

* 好處：這能幫助大語言模型（LLM）極其精準地識別「指令」與「參考資料」的邊界，大幅提升生成準確度。  

<br>

## 3. 實戰操作流程拆解

### 第一步：建立原始規格書（SPEC.md）
1. 先在 ChatGPT 中以白話文描述專案需求，請其生成一份結構完整的「技術規格書」。  
2. 在專案根目錄建立 `SPEC.md` 檔案，並將內容貼上（此時格式可能較為混亂）。  

### 第二步：餵入指令與外部脈絡，啟動 Claude Code

CLI：

```Plaintext
We're building an app described in @spec.md . 
Please format this file as proper Markdown.

Also update the file and update the part about the users table and auth-related tables. We're using the betterauth library, which expects a certain database structure:

<betterauth_docs>
[此處貼上從 Better-Auth 官網複製的 Markdown 資料庫 Schema 文件]
</betterauth_docs>
```

### 第三步：切換加速模式並執行
在送出提示詞前，按下快捷鍵 SHIFT + TAB 切換到 「自動接受修改模式（Accept Edits Mode）」，然後按下 Enter。Claude Code 隨即自動讀取 Spec.md、消化 Better-Auth 的文件，並完美重構了整份規格書。

### 知識圖解：規格書重構前後對比
```
[原始 ChatGPT 規格書] ───> (偵測到 Users 複數資料表，與 Better-Auth 衝突)
                                      │
                                      ▼ <加上 Better-Auth 官方文件脈絡>
[Claude Code 自動重構] ───> 1. 修正為 "user" 單數資料表與對應欄位
                            2. 完成精美的 Markdown 格式化
```

# Note: Prompt & Content Engineering Recommendation

## 1. 簡潔且精準（Concise & Precise）
* **核心心法：** 保持提示詞聚焦、乾淨。 
* **實作要點：** 你必須準確且清晰地描述希望 AI 解決的任務或問題，但請絕對避免塞入無關緊要的細節、廢話或無意義的客套修飾（Unnecessary Fluff）。  

## 2. 拒絕不必要的上下文（No Unnecessary Context）
* **核心心法：** 提供有用（Useful）的脈絡至關重要，但塞入無關的資訊只會適得其反。 
* **實作要點：**
    * 不要瞎猜檔案：只用 @ 標記你確定（KNOW）相關的檔案；不要因為「覺得（THINK）某個檔案可能有關係」就盲目把它 @ 進去。 
    * 過濾外部資料：無論是官方文件、報錯 Log 還是其他參考資訊，只要與當前任務沒有直接關係，一律不要提供，以免干擾 AI 的注意力。  

## 3. 先思考、再規劃、後提問（Think, Plan, Prompt）
* **核心心法：** 不要一邊敲鍵盤一邊想，更不要指望「等 AI 改錯了再來慢慢修」。
* **實作要點：** 
    * 動手前，先在腦中思考、建立清晰的實作計畫，最後才撰寫提示詞。
    * 如果你發現自己經常需要連續發送好幾條提示詞來修正、澄清 AI 的行為，這代表你前置的規劃不夠好。 
    * 對於所有非微不足道（Non-trivial）的複雜任務，請強烈建議啟動 「計畫模式（Plan Mode）」。  

## 4. 絕對不要「測試」AI（Don't "Test" The AI）
* **核心心法：** 你是要叫 AI 來幫你把工作搞定的，不要跟它玩捉迷藏。 
* **實作要點：** 
    * 如果你事先就知道這項任務存在某個技術難關、特定地雷或常見的踩坑點（Pitfall），請直接向 AI 攤牌，絕對不要對它隱瞞資訊。  
    * 在初始提示詞中，應直接指出已知的潛在錯誤，並附上你推薦的解決方案（Recommended Solution），引導 AI 直接繞過陷阱。  

## 5. 明確指定 AI 應該使用的工具（Explicitly Guide Tool Usage）
* **核心心法：** 不要一廂情願地「希望」AI 會自動聯想到正確的工具。
* **實作要點：** 
    * Claude Code 擁有強大的工具庫（例如底層的 Bash 執行、網頁請求，或透過 MCP 伺服器擴充的工具、子代理人 Sub-agents、特化技能等）。 
    * 如果你清楚知道某個任務（或任務的某個步驟）使用特定的工具或功能來做最合適，請在提示詞中明確下達指令指示它使用。

# 自主套件管理與 Claude Code 專案初始化（/init）

### 一、 為什麼「套件安裝」要親自動手？

- **避免 AI 的「時空錯亂」：**
AI 在自動安裝套件時，有時會傾向直接去修改 package.json 檔案，並在裡面填入舊版本的套件號碼，這會為專案埋下技術債。

- **確保最新穩定版本：**
身為開發者，你清楚知道目前需要哪些套件，親自使用套件管理工具（如 bun add）能 100% 保證安裝到最新、最正確的版本。

範例安裝的套件清單：
1. **better-auth：** 第三方身份驗證核心。  
2. **zod：** 用於嚴格驗證使用者輸入（User InputValidation）。  
3. **Tiptap 相關套件：** 遵循官方指南安裝富文本編輯器所需的依賴。  
4. **bun-types（開發依賴 -D）：** 避免 TypeScript 在處理 Bun 原生 API 時噴出型別錯誤。

<br>

### 二、 核心指令： `/init`

**CLI**

```Plaintext
/init
```

#### 1. `. /init` 到底做了什麼？

不論是全新的專案還是第一次引入 Claude Code 的既有老舊專案，`/init` 都會觸發 AI 對整棟代碼建築進行「深度安全檢查」與「架構通靈」：  

- 依賴分析：徹底掃描剛剛手動安裝的所有套件與版本。  
- 結構理解：翻遍專案內的所有資料夾與檔案，理解目前的架構設計。  
- 規格對齊：自動偵測並讀取專案中的 Spec.md，將技術規格與實際代碼進行交叉比對。  

#### 2. 最終產物：CLAUDE.MD 誕生

- 經過深度分析（期間會向你請求資料夾的探索權限），Claude Code 會在專案根目錄自動生成一個名為 CLAUDE.MD 的檔案。

# 深入解析 CLAUDE.md：專案的核心控制器與長期記憶

`CLAUDE.md` 是任何 Claude Code 專案中最重要的核心檔案。它的核心目的，是為 AI 提供跨越不同工作階段（Sessions）的長期記憶（Long-term memory），並用來規範 AI 必須遵守的全局開發準則。

## 一、 `CLAUDE.md` 的三大核心運作原理

* **自動載入機制：** 每當你啟動一個全新的 Claude Code 工作階段，或是在對話中清空上下文（Clear context）時，系統都會自動且強制地將 `CLAUDE.md` 的內容加載到新建立的對話大腦中。  

* **保持短小精悍（Slim & Simple）：** 這份檔案必須刻意維持精簡。因為它作為提示詞的一部分，會隨時佔用你的 Token 額度，如果塞入太多繁雜資訊，不僅會浪費成本，還會污染（Pollute）對話上下文。  

* **分層巢狀載入（Nested Files）：** 專案中不限於只能有一份 `CLAUDE.md`，你可以根據需求在 subfolders 中加入任意層級的巢狀檔案：  
    * 根目錄的 `CLAUDE.md`：適用於全專案的全局通用規則（例如指出專案使用 Bun，並要求 AI 使用 Bun 來執行專案腳本）。  
    
    * 子資料夾中的 `CLAUDE.md`：例如在 `app` 資料夾下另外放一份 `CLAUDE.md`。只有當 Claude Code 實際動手修改或讀取該子資料夾內的檔案時，才會動態載入並考慮這份子檔案的內容。  

<br>

## 二、 實戰為 `CLAUDE.md` 注入黃金指令

在生成的初始檔案中，手動加入了兩個極其關鍵的進階微調：

### 1. 建立與規格書的「條件式關聯」

為了讓 AI 理解專案藍圖，但又不想讓它每次都強行讀取整份龐大的 spec.md 造成 Token 浪費，講師加入了以下提示：

```Paintext
We're building the app described in @spec.md
```

並給予 AI 明確的提示（Hint），要求它只有在處理全局架構任務、或是需要連帶確認（Double-check）資料庫結構、技術堆疊或整體應用程式架構時，才去翻閱 `@spec.md`。

### 2. 強制實施「省錢省空間的簡短回覆令」

AI 有時會過度熱情，吐出長篇大論和大量的程式碼片段。這不僅會增加使用者付費的 Token 成本，也會隨著對話拉長而污染上下文。因此講師加入了核心限制：

```Paintext
Give me short replies, no long essays with lots of code snippets in there.
```

效果：強制 Claude Code 有話直說、精簡回覆，把每一顆 Token 都花在刀口上！

<br>

## 三、開發者必備的配套動作：環境同步

為了確保 CLAUDE.md 中規定的「全專案使用 Bun 運行」能徹底落實，並順利調用 Bun APIs，順手修改了專案的 package.json：

```json
// package.json 變更範例
"scripts": {
  "dev": "bun run --bun next dev",
  "build": "bun run --bun next build",
  "start": "bun run --bun next start"
}
```

原因：加上 `--bun` 旗標能確保開發伺服器（Dev server）、打包（Build）與啟動（Start）命令完全運行在 Bun 的 Runtime 之上，完美呼應 `CLAUDE.md` 中的環境規範。

```
專案根目錄/
├── CLAUDE.md  ──────> [永遠自動載入] 規範全局使用 Bun、簡短回覆、何時看 spec.md
├── spec.md
├── package.json
└── app/
    ├── CLAUDE.md  ──> [動態載入] 只有當 AI 修改 app/ 內的檔案時，才會疊加此處的規則
    └── page.tsx
```

# 實戰：啟動 Plan Mode 與分頁路由（Routes）初始化

最重要的開發流程 —— 「計畫模式（Plan Mode）」與對話審查循環。
不論任務看起來多簡單，善用 Plan Mode 都能大幅提高程式碼的一次過關率。

## 一、 首發任務：建立核心分頁路由

為了降低複雜度，講師將第一個任務限縮在極小的範圍（Focused Task）：

* 目標：建立專案所需的所有核心頁面與路由（Routes）。
* 限制條件：先不實作真實的身份驗證、不寫入正式的頁面內容，各分頁只需放入測試訊息（Dummy message）即可。

在送出 Prompt 之前，使用 SHIFT + TAB 快捷鍵將模式切換至 「計畫模式（Plan Mode）」。

<br>

## 二、核心機制：為什麼你該預設使用「Plan Mode」？

除了極度微不足道的簡單修改外，其餘所有任務都應該預設從 Plan Mode 開始。

### 1. Plan Mode 的運作原理

* **只規劃，不動手：** 在此模式下，Claude Code 完全沒有修改檔案的權限。它會先深入探索你的專案架構，並針對你的任務產出一份「實作計畫藍圖」。

* **把平庸的提示詞變優秀：** 如果你的 Prompt 寫得不夠完美、或任務太複雜，AI 會在這個階段主動向你提出澄清問題（Clarification questions），引導你補齊規格。

* **本地臨時記憶保護：** 生成的計畫會臨時儲存在本地端。即使後續對話太長導致上下文視窗（Context Window）爆滿、需要執行 /clear 清空時，AI 依然能夠回頭讀取這份計畫，記憶絕不丟失。

### 2. 審查計畫與互動選項

當 Claude Code 吐出計畫後，通常會提供數個互動選項供開發者選擇（例如：清除對話並自動接受修改、保留對話並手動確認修改、或是給予更多回饋以修正計畫）。

<br>

## 三、實戰：動態修正計畫與執行

在 AI 動手改 code 前，及時校正航向：

```Plaintext
I only want a single /authenticate route which supports only email plus password auth.
（我只要一個單一的 /authenticate 路由，且只支援信箱加密碼驗證。）
```

**1. 計畫修訂：** 送出上述微調後，Claude Code 不會直接去寫 code，而是回到 Plan Mode 重新修正並確認它的實作藍圖。

**2. 授權與執行：** 確認計畫完美後，講師選擇了「清空上下文並自動接受所有修改（Clear context and auto-accept all edits）」。

* 註：清空上下文能讓 AI 帶著最乾淨、無雜訊的大腦去執行寫 code 任務，避免前面的聊天雜訊干擾生成品質。

**3. 自動化驗證機制：** 寫完程式碼後，Claude Code 會主動請求執行 Linting（語法檢查）與 Build（專案打包構建），確保它剛剛寫的程式碼 100% 能夠順利運行。

<br>


## 五、總結：

不要把 Claude Code 當成盲目敲鍵盤的機器，要把它當成你的架構副手。在 Plan Mode 階段把規則、路由名稱、驗證方式對齊（例如示範即時修正單一 /authenticate 路由），後面 AI 產出的 code 就會精準得讓人驚嘆，完全不需要耗費時間在後續重複修補！

# 補充： Claude Code 的三種操作模式？

|模式|英文名稱|行為特徵|適合情境|
|--|--|--|--|
|詢問模式|Ask|Mode|純對話，不執行、不寫檔|問問題、釐清概念|
|規劃模式|Plan|Mode|列出步驟讓你確認，唯讀|多步驟任務、複雜變更|
|執行模式|Code|Mode|直接執行，有完整操作權限|明確的開發與自動化任務|

<br>

```
模式選擇流程圖：

你的任務是什麼？
    │
    ├── 只是想問問題 / 釐清概念
    │        └──▶  Ask 模式（純對話）
    │
    ├── 有多個步驟，不確定 AI 會怎麼做
    │        └──▶  Plan 模式（先列計畫再確認）
    │
    └── 目標明確，直接讓 AI 動手
             └──▶  Code 模式（全自動執行）
```

## 名詞解釋

* **Ask 模式：**
Ask 模式是最安全的模式，Claude 只能回答問題、分析程式碼，不會執行任何指令、不會動任何檔案。就像在跟一個非常聰明的助理聊天，他可以給你建議，但什麼都不會自己動手。適合：初次探索新工具、問架構設計問題、釐清需求。

* **Plan 模式：**
Plan 模式是 Claude Code 的招牌功能。進入 Plan 模式後，Claude 會先用「唯讀」方式分析你的程式庫或任務，列出完整的執行計畫，並等你確認後才繼續。這段期間它讀得了檔案、搜尋得了程式碼，但絕對不會修改任何東西。適合：複雜的多步驟任務、不確定 AI 會怎麼做的情況。

* **Code 模式：**
Code 模式是完整執行模式，Claude 有權限讀取、寫入、執行指令。適合目標明確、你已經知道要做什麼的任務。在這個模式下 Claude 動作最快，但也最需要你有清楚的指示。

<br>

## 切換方法

### 1. CLI 下 Shift + Tab 切換模式
* 預期畫面：底部模式指示器依序切換：

    Code → Plan → Ask → Code…

* Plan 模式顯示：⏸ plan mode

### 2. 確認目前模式後，直接輸入你的任務描述
* Ask 模式：Claude 只會回答，不執行
* Plan 模式：Claude 列出步驟等你按 Enter 確認
* Code 模式：Claude 直接執行

### 3. 從命令列直接指定模式啟動

```
# 以 Plan 模式啟動（適合複雜任務前的規劃）
claude --permission-mode plan

# 以 Ask 模式啟動（適合純諮詢）
claude --permission-mode ask

# 以 Code 模式啟動（預設，適合明確執行）
claude
```

# 深入解析 Claude Code 幕後功臣：自動化子代理人（Sub-agents）

在進行複雜的代碼分析或全面審查時，你會發現終端機畫面上同時有多個任務在並行運作。這正是 Claude Code 核心的「多代理人架構」在發揮威力。

## 一、 子代理人（Sub-agents）的運作原理

當你下達一個複雜的綜合指令（例如：同時去查最新 `Better-Auth` 文件，並比對目前專案的程式碼是否合規）時，Claude Code 不會單靠一個大腦做完所有事，而是會像專案經理一樣將任務拆解，並發包給「內建的子代理人」。

* 並行處理（Parallel Processing）：主代理人（Main Agent）一邊利用 Context7 MCP 瘋狂查閱官方文件，另一邊則同時派遣子代理人分頭去執行其他任務（例如掃描資料夾結構）。

* 特化專家模式：內建的 Explore Agent（探索代理人） 就是一個標準的例子。它是一個被極度優化、專門用來快速閱讀、拆解並理解檔案與目錄結構的特化型 AI。

<br>

## 二、核心優勢：為什麼要使用子代理人？

子代理人除了能加速執行之外，最關鍵的工程原因在於：防止主對話的「上下文污染（Context Pollution）」。

### 1. 獨立的上下文視窗（Isolated Context Window）：
子代理人擁有自己獨立的大腦空間。它在翻閱你專案中幾十個檔案、幾百行程式碼時所產生的巨大 Token 消耗，完全不會累積到你的主對話視窗中。

### 2. 只匯報精華結果：
子代理人任務結束後，只會將「高度壓縮的結論或摘要」回傳給主代理人。

### 3. 終端機透明化標示：
在畫面上，只要看到帶有 Token 消耗數量（Tokens consumed）以及執行耗時（Time taken）的獨立區塊，那就代表該任務是由子代理人獨立關起門來完成的。

<br><br>

# 打造專屬「自訂子代理人（Custom Sub-agent）」

雖然主代理人可以自行查閱文件，但逐條在主視窗中讀取長篇大論會瞬間撐爆上下文空間。透過建立一個專門的 DocsExplorer（文件探索專家），我們可以將高額的 Token 消耗徹底隔離在外。

## 一、 自訂子代理人的檔案配置架構

Claude Code 允許開發者在專案本地（Local）或系統全域（Global）建立特化型的 AI 子兵團。檔案必須採用 Markdown (.md) 格式，且存放的路徑結構有嚴格的命名限制：

### 專案本地（僅限當前專案使用）：
放置於 .claude/agents/ 資料夾下。(註：agents 資料夾名稱為官方強制規定，不可自訂)。

### 全域系統（所有專案皆可調用）：
在 macOS/Linux 環境下，放置於家目錄的 ~/.claude/agents/ 內。

```Plaintext
專案根目錄/
├── .claude/
│   └── agents/            <─── [強制規定名稱]
│       └── DocsExplorer.md <─── [自訂代理人檔案]
├── CLAUDE.md
└── spec.md
```

<br>

## 二、 核心解密：DocsExplorer.md 的藍圖設定

一份標準的自訂代理人 Markdown 檔案，包含了四個核心要素：名字（Name）、描述（Description）、工具授權（Tools）與模型指定（Model）。

以下為講師所設計的 DocsExplorer.md 範本結構：

```Markdown
# DocsExplorer

## Description
一個專門用來深度尋找、閱讀與分析各類前端/後端程式庫（Libraries）、框架（Frameworks）官方文件的特化型子代理人。

## Tools
- mcp-search # 強制授權：允許此子代理人動態載入與調用 MCP 工具 (例如 Context7)

## Model
claude-3-5-sonnet # 指定模型：文件探索不需要最頂級的推理，選擇速度快、性價比高的 Sonnet (或 Haiku) 即可

```

<br>

## 三、 子代理人的內在靈魂：高階提示詞工程（Instructions）

在 Markdown 的內文中，講師使用極具開發者智慧的「白話文」來規範這個子代理人的工作流程與行為準則，包含以下三大黃金指令：

### 1. 強制並行處理（Parallel Execution）

要求子代理人在查閱多個套件（如同時查 Better-Auth 與 Bun SQLite）時，必須同時、並行發起檢索，以極大化提升執行速度。

### 2. 多層級阻斷式搜尋策略（Fallback Mechanism）
* 第一優先：使用 Context7 MCP 伺服器 作為最主要的官方知識來源。
* 第二優先（備援機制）：若 Context7 找不到，才允許啟動 Web Search（網路搜尋）。

### 3. LLM 友善優先原則（LLM-Friendly First）
指示子代理人在進行網頁搜尋時，不要盲目抓取雜亂的 HTML 網頁。必須優先尋找以下兩類資源：

* 目標網站是否有提供 llms.txt 檔案（現代網站專門餵給 AI 讀取的精簡文檔總綱）。

* 目標網站是否有直接提供官方的 .md（Markdown）原始文件檔。

* 只有在上述兩者都找不到的情況下，才可以去爬一般的網頁內容。

<br>

## 四、子代理人的啟動與觸發
當你在新 Session 中重複相同的審查任務時，Claude Code 會讀取 DocsExplorer.md 中的 Description（描述）。一旦 AI 判定當前任務涉及「查閱框架文件」，它就會自動把任務打包，丟給 DocsExplorer 去閉門造車，最後只把乾淨的結論收回到主視窗中！

```
我們能做到的，絕對不止是用猜的或『希望』它會自己去用這個代理人
```

## 一、 100% 觸發：在 CLAUDE.md 寫入調度鋼鐵律令

為了確保 Claude Code 在每一次處理第三方套件時，都能自動且強制使用我們建立的 DocsExplorer 子代理人，講師在根目錄的 `CLAUDE.md` 中加入了以下黃金規則

* **優化後（CLAUDE.md 強制發包）：**主視窗變得極度乾淨！Claude Code 讀到規則後，同時在幕後一口氣複製出多個 DocsExplorer 子代理人實例，一邊並行去啃 Better-Auth 文件，一邊去啃 Bun SQLite 文件，主代理人只負責接收最終的精簡摘要。

## 二、 實戰除錯：AI 聯網審查抓出 Better-Auth 致命盲點

在這次經由子代理人加持的深度審查中，Claude Code 雖然之前寫出了堪稱完美的骨幹，但經過最新 Next.js 官方整合指南的比對，成功抓出了一個嚴重的遺漏：

### 發現的問題：遺漏 Next.js Cookie 插件
在 Next.js (App Router) 環境下整合 Better-Auth，除了後端萬用路由要使用 toNextJsHandler 之外，為了讓前端與伺服器端元件（Server Components）能正確讀取登入狀態，必須在後端設定中注入 nextCookies 插件。

既然 AI 已經精準幫我們通靈出問題點，這時候身為開發者，為了節省 Token 成本，不一定要叫 AI 改，自己動手補上幾行 code 反而是最聰明且省錢的作法。

優秀的 AI 開發者會隨時在「讓 AI 代勞」與「自己動手（DIY）」之間尋找成本與速度的最佳平衡點。利用子代理人幫我們做大面積的技術審查與抓漏，再由我們親手補上關鍵程式碼，就是最高段、最省錢且威力驚人的現代化人機協作完全體！

# 深入解析 Agent Skills（代理人技能）：跨工具的 AI 知識外掛標準

Agent Skills 是一個跨 AI 程式碼工具（如 Claude Code, Cursor, OpenCode 等）支援的開放標準。它的核心思想是：動態載入一整套與特定任務相關的「上下文、規範與工具鏈」，在需要時為 AI 提供即時的專家指導。

## 一、 代理人技能（Agent Skills）的組成結構

一個完整的「技能包」通常由以下幾個部分組合而成，結構高度彈性：

### 1. skill.md（必備核心）
這是技能的靈魂檔案。裡面用 Markdown 定義了這個技能的名稱、觸發時機、以及核心的使用說明與規範。

### 2. 附加文件與資料夾（選配）
如果該技能需要大量的背景知識（例如：公司內部的 React 元件架構白皮書、特定演算法的設計範本），可以將這些 .md 或文字檔與 skill.md 放在一起，供 AI 動態查閱。

### 3. 自動化腳本 Scripts（選配）
進階的技能甚至可以賦予 AI 執行特定腳本的權限（可用 Python 或 JavaScript 撰寫）。例如：一個「程式碼瘦身技能」可以附帶一個自動刪除未引用檔案與清理快取的專用 Clean-up 腳本。

### 4.多媒體資產 Assets（選配）
可以包含圖片、架構圖表（Diagrams）等，讓 AI 在執行任務時能連帶參考視覺化的設計藍圖。

<br>

## 二、 開發者實戰經驗：最實用的「知識型技能」
雖然技能可以綁定複雜的腳本，在實務開發應用程式時，最實用、最高效的往往是「知識型技能（Knowledge Skills）」。

這些技能不搞花哨的自動化腳本，而是專注於提供：

* **最佳實踐（Best Practices）：** 例如「如何撰寫極致效能的 React Hooks」或「如何設計防禦性的 Python 類別」。

* **團隊編碼規範（Coding Standards）：** 強制 AI 在特定任務下遵循團隊專屬的命名與架構邏輯。

* **特定框架的踩坑指南：** 將過去團隊踩過的技術雷區整理成冊，變成 AI 的內建抗體。

<br>

## 三、知識圖解：Skills 的動態載入機制
不同於 CLAUDE.md 這種無差別、每場 Session 都強制載入的全局規則，Agent Skills 是「動態（Dynamically）載入」的。

```Plaintext
[ 開發者下達指令：請幫我重構此 React 元件 ]
                           │
                           ▼ (AI 辨識任務性質)
  ┌──────────────────────────────────────────────────┐
  │ 偵測到關鍵字 ──> 自動加載 【React_Component 技能包】 │
  │ 包含：                                            │
  │  ├── skill.md (觸發條件與核心指南)                 │
  │  └── best-practices.md (Hooks 效能優化白皮書)      │
  └────────────────────────┬─────────────────────────┘
                           │
                           ▼ (帶著專屬外掛知識)
         [ AI 產出完全符合團隊高標準的完美程式碼 ]
```

如果說 CLAUDE.md 是專案的「基本憲法」，Agent Skills 就是專案的「專家工具箱」。當你需要 AI 寫 React 時，它就帶上 React 箱；需要它寫 SQL 時，它就切換成 DBA 箱。

# 打造 React 專家技能包（Skills）與動態上下文工程

為了讓 Claude Code 寫出最高規格的 `React` / `Next.js` 原始碼，我們不能把所有的規則都塞進 `CLAUDE.md`，否則會撐爆 Token 空間。最完美的做法是建立「動態加載」的本地技能包。

## 一、 技能包（Skills）的嚴格目錄規範

與子代理人相同，Claude Code 對於技能包的資料夾與檔案結構有著鋼鐵般的硬性規定：

* **資料夾命名：** 必須在 .claude/ 底下建立一個名為 skills 的子資料夾。

* **技能子目錄：** 在 skills/ 內，每一個技能都要有自己獨立的資料夾。該資料夾的名稱就是這個技能的唯一識別碼（ID）。

* **核心檔案：** 每個技能資料夾內，必須包含一個完全符合大小寫的 `Skill.md` 檔案。

```Plaintext
專案根目錄/
└── .claude/
    └── skills/                <─── [強制規定名稱]
        └── modern-best-practice-react-components/  <─── [技能唯一識別名稱]
            ├── Skill.md       <─── [強制規定名稱，核心元數據]
            └── references/    <─── [慣例命名：存放超大體量的延伸補充文件]
                └── YouDon'tNeedUseEffect.md <─── [特化地雷區指南]
```

<br>

## 二、 解密 Skill.md 元數據與「漸進式加載

在 Skill.md 的頂部，有兩個欄位是官方強制必填的：Name（必須與資料夾名稱完全相同） 與 Description（描述）。

### 1. 它是如何做到不污染（Pollute）上下文的？
當你啟動 Claude Code 時，系統只會把所有技能的「Name」與「Description」載入到大腦中。

只有當 AI 判定當前的任務（例如：建立 /authenticate 登入頁面元件）與某個技能的描述高度相關時，它才會在幕自動把該技能 Skill.md 的完整內文讀入上下文視窗。

### 2. 終極省 Token 技巧：references/ 盲盒機制

React 的 useEffect Hook 極其複雜且容易寫出記憶體洩漏（Memory Leaks）。為了不讓這些沉重的範例程式碼天天佔用 Token，使用了進階技巧：

* 在技能中建立 `references/` 資料夾，並放入 `YouDon'tNeedUseEffect.md`。

* 在 Skill.md 內留下一句話導引 AI："If you are working with useEffect, look at the files in the references folder."

* 效果：動態發現（Dynamic Discoverability）的極致！AI 只有在發現自己要寫 useEffect 時，才會去打開 references/ 盲盒，平時完全不佔空間。

### 3. 本地（Local）與全域（Global）技能的抉擇：
雖然技能可以像子代理人一樣放在家目錄`（~/.claude/skills/）`變成全域共享，但講師強烈建議使用本地專案專屬技能。因為不同專案的技術棧可能完全不同（例如有些專案根本不用 `Next.js`），全域加載會讓 AI 的大腦載入無關的 Name 與 Description，造成不必要的浪費。

<br>

## 三、 實戰演練：建立真實驗證頁面與「執行中提示注入」

在 AI 進入 Plan Mode 與實際撰寫的過程中，講師展示了兩個極高階的實戰手法：

### 1. 拒絕計畫並修正盲點
AI 在最初的計畫中，打算在註冊時收集使用者的「姓名（Name）」。講師即時攔截並予以拒絕（Deny），強制要求："I just want to use their email for that, no name input."，引導 AI 修正藍圖後才授權執行。

### 2. 運行中即時提示注入（Message Injection）
在 Claude Code 正在後台瘋狂寫 code 的時候，講師直接在終端機輸入並送出了一條新提示：

```
Use modern, clean, modern JSX, and Tailwind code.
```

* **運作機制：** 當 AI 正在埋頭苦幹時，你輸入的訊息不會打斷它，而是會被暫存並在下一個合適的空檔「注入」到 AI 的思考流中。這能極大化確保 AI 在編碼的最後一刻，依然緊緊綁定我們在技能包中所規範的 Tailwind 與現代 JSX 風格。

# 解密自訂技能延伸的斜線指令（Slash Commands）

在 Claude Code 終端機中輸入 / 時，你會看到一個包含「內建指令」與「自訂技能」的綜合選單。理解這些指令的觸發邏輯，能幫你省下許多無謂的 Token 浪費。

## 一、 終端機中的斜線指令三大分類

目前你在選單中看到的指令，主要可以分為以下三類：

### 1. 核心內建指令（Built-in Commands）
系統自帶的功能，用於控制工作階段或專案狀態。

* `/init`：深度分析專案架構並生成 CLAUDE.md。

* `/config`：檢視並微調 Claude Code 的核心環境設定。

* `/clear`：徹底清空當前對話大腦，還原乾淨的上下文視窗。

### 2. 技能衍生指令（Skill-based Commands）

當你在 .claude/skills/ 資料夾下建立了任何自訂技能（例如包含 Skill.md 的資料夾），該技能的名稱就會自動同步變成一個可手動輸入的斜線指令。

### 3. 客製化執行指令（Custom Commands）

這是專門設計來讓你主動輸入、命令 AI 去執行特定自動化動作的專屬指令。

<br>

## 二、核心心法：技能衍生指令該手動執行嗎？
當看到自己寫的技能（例如 /bun-instead-of-node）出現在選單中時，很多開發者的直覺是去敲擊它。非常關鍵的實戰提醒：

```
「技能（Skills）的預設核心價值是讓 AI 自動通靈（Discover），而不是由你來手動呼叫。」
```

### 1. 為什麼對著「知識型技能」敲斜線指令會失敗？
如果你的技能純粹是「知識型技能（Knowledge Skills）」（例如紀錄了團隊的 React 最佳實踐、或是使用 Bun 代替 Node 的規範），你單獨輸入該指令（如執行 /bun-instead-of-node），AI 運作後會回答你它不知道要做什麼。

* 原因：因為這類檔案裡只有滿滿的「背景知識」，你手動敲了指令，卻沒有給它任何具體的任務目標（例如：重構哪段 code），它自然會不知所措。

### 2. 什麼時候才適合手動呼叫技能指令？
只有當你的技能包裡面附帶了具體的執行指令（Extra Instructions）或自動化腳本（Scripts）時，手動呼叫才有意義。例如：

* 一個名為 `/cleanup` 的技能，裡面綁定了自動清理快取與編譯暫存檔的 Python 腳本，這時你手動輸入它，就能完美觸發自動化任務。

# 技能優化、參數化（$ARGUMENTS）與第三方技能生態系

## 一、 技能大腦微調：如何引導「自動發現機制」？

如果你發現自己寫的自訂技能常常被 Claude Code 冷落（沒有自動動態加載），問題通常出在 Skill.md 的 Title（標題） 與 Description（描述）。因為 AI 在初始階段只看這兩個欄位來決定要不要翻牌。

* 強烈觸發字眼：在 Description 中加入極度具體且帶有強烈暗示的指令。
    * 錯誤示範：React component tips.（太模糊，AI 很難判定何時該用）。

    * 正確示範：Use this skill to optimize React components and check for the usage of common best practices. 或 Gets triggered when writing ANY kind of CSS code.

<br>

## 二、 鋼鐵控管：進階元數據（Metadata）配置

透過在 Skill.md 的 YAML 區塊中加入以下高階參數，你可以完全扭轉技能的運作行為：

|進階參數|設定值|實戰效果（背後原理）|
|--|--|--|
|disable-model-invocation|true|關閉 AI 自動通靈。這個技能「不會」被 AI 自動加載，只能由你在終端機手動輸入 /技能名 當作自訂指令執行。|
|user-invocable|false|禁止人類手動呼叫。這個技能「不會」出現在你的斜線指令選單中，100% 只能由 AI 根據任務在後台自動探索並加載。|
|allowed-tools|Read|限制 AI 的破壞力。例如設定為 Read，代表 AI 載入此技能時只能看、不能動（無法修改或建立檔案），非常適合用來做純代碼審查。|

<br>

## 三、 進階實戰：打造帶有動態參數的 /code-review 技能

展示一個極具啟發性的技巧：利用特殊的 `$ARGUMENTS` 預留字，讓技能可以接收你在終端機輸入的動態參數！

### 檔案建立：`.claude/skills/code-review/SKILL.md`

```
---
name: code-review
description: Review code for bugs, security or performance issues. Use this skill when asked to perform code reviews or after finishing major tasks or after refactoring code.
allowed-tools: Read
---

MODE: $ARGUMENTS

If Mode is one of the following, adjust the review as described:
- MODE == BUGS: Focus ONLY on logical or other bugs.
- MODE == SECURITY: Focus ONLY on security issues.
- MODE == PERFORMANCE: Focus ONLY on performance issues.

If MODE is set to anything else or nothing at all, perform a thorough, general code review.

Carefully and thoroughly explore the codebase file-by-file to find potential issues, create a detailed report. Don't rush it!
```

### 手動呼叫：

當你在終端機輸入 /code-review SECURITY 時，$ARGUMENTS 就會動態變成 SECURITY！Claude Code 會自動進入這個技能，且因為 allowed-tools: Read 的安全限制，它絕對不會手抖去亂改你的程式碼，只會安分地幫你進行深入的安全漏洞掃描並吐出報告。

<br>

## 四、 借力使力：引入第三方免費技能寶庫（skills.sh）

開發者不需要總是自己從零造輪子。目前網路上有一個非常熱門、且完全免費的公共 AI 技能倉庫 —— `skills.sh`。

這裡面累積了來自全世界開源社群針對各種程式語言、框架優化、資安防護所撰寫的高階技能包。

### 1. 安裝第三方技能（需要 Node.js 環境）
你可以直接在專案終端機中執行以下指令，一鍵將特定社群技能下載並安裝到你的專案中：
```
npx skills add <擁有者名稱/倉庫名稱>
```

<br>

### 2. 開發者完全掌控權
這套機制的設計非常體貼開發者。透過 npx 下載下來的技能，會直接以實體 Markdown 檔案的形式落戶在你的專案資料夾中。
* 這意味著你擁有 100% 的主導權！你可以隨時打開這些下載下來的 .md 檔案進行微調（Tweak）、刪除不適用團隊的規則，或是根據自身專案需求進行二次魔改。

### Review

```
  │
  └── [特化領域技能] ──────> 放入 .claude/skills/ 資料夾
        │
        ├── 想自己寫 ────> 建立 Skill.md (活用 $ARGUMENTS 與 allowed-tools)
```
